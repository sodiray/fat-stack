Give a new project a fresh start by producing the three foundation documents fat-stack needs to iterate: a product overview, a technical stack doc, and a patterns doc. Do not ask for confirmation; begin immediately.

## What fresh-start produces

Three files, nothing more:

- `docs/product/overview.md` — what the product is, who it's for, how they use it.
- `docs/technical/stack.md` — the stack, tooling, one-paragraph architecture.
- `docs/technical/patterns/core-rules.md` — the hard rules `/dev` follows when writing code.

Fresh-start does not write feature docs, subsystem designs, or any code. Those come from the other skills running on their normal cadence.

## Open with a single paragraph of framing

Before the first question, tell the user something like:

> "I'm going to walk you through three foundation docs. I'll ask a small set of questions about the product, propose a stack based on your answers, and give you a starter set of coding rules you can edit. Then you're set up. If you want to know what to do next at any point, type `/fat-help`."


Do not list the questions in advance. Do not estimate how long this takes. Move on.

## Part 1 — Product (3 questions)

Ask one question at a time. Keep answers short.

1. **What are you building?** Accept the answer, then restate in one sentence: "So, this is a ___ for ___ that ___. Right?" The agreed sentence becomes the top of the product doc.

2. **Who will be using it?** Push back once if the answer is a generic role. "Developers" is not an answer — "solo developers shipping side projects on weekends" is. Stop pushing after one correction.

3. **How do they use it — what's the basic experience?** One short paragraph is plenty. You're looking for the core workflow in the user's own words.

When you have those three answers, stop. Do not go deeper into capabilities, states, rationale, or scope — those come later from `/product-author`.

## Part 2 — Technical (3 questions, then propose)

Ask:

1. **How will people interact with it?** CLI, web app, mobile app, API, library, browser extension, desktop app, something else.

2. **How many users today, and roughly where does that go?** 1 / 10s / 100s / 1000s / 10000s+. This shapes database, hosting, and infra choices.

3. **Any hard constraints?** Languages known/not-known, required platforms, offline requirements, no paid services until revenue, organizational constraints.

**Then propose a coherent stack in one message.** Based on the three answers, recommend:

- Language and runtime
- Package manager
- Test framework
- Linter / formatter
- Two or three core libraries appropriate to what they're building (e.g., for a CLI: an arg parser; for a web app: a framework; for an API: an HTTP framework)
- External services the project will need (database, auth, payments) — pick defaults or say "none needed"

Each choice gets one sentence of reasoning. End with: "Want this, or change something?"

The user confirms or amends. Iterate once if they want changes. Do not loop forever. If they can't decide, pick and say: "Going with X — you can change it later, the patterns doc stays stable."

## Part 3 — Patterns (starter, then customize)

Show this starter set inline:

> **core-rules.md starter:**
> 1. Prefer immutable locals. If a value changes, derive a new one via a function rather than reassigning.
> 2. Prefer small functions and plain data over classes. Reach for a class only when identity and state lifecycle matter together.
> 3. Prefer guard clauses and early returns over deep conditional nesting.
> 4. Validate external input at the system boundary with runtime checks. Trust internal data within a module.
> 5. Avoid speculative generality. Build the concrete case. Abstract once you have three examples, not one.
> 6. No opaque type escapes (`any`, untyped parameters, etc.) without a comment explaining why.
> 7. Keep public interfaces explicit. Every exported function declares its parameter and return types.
> 8. Tests import the unit under test directly; avoid mocking collaborators within the same module.
> 9. Delete unused exports. If nothing consumes it, remove it.
> 10. Comments explain *why*, not *what*. If a comment describes what the code does, rewrite the code instead.

Ask: "Any you want off, anything to add?"

Accept edits. Don't negotiate — the user knows their preferences. Lock the list.

## Part 4 — Write the three docs

Write all three in one pass. Use the fat-docs MCP tools (`search_docs`/`index_docs`).

### `docs/product/overview.md`

YAML front matter: `title` and `summary` only. (Groups come from `docs/groupings.md`, which doesn't exist yet on a fresh project — it's generated by the `groupings` skill once the doc corpus has grown. Skip it for now.) Under 150 lines.

Sections:
- **What this is.** The one-sentence description from Question 1.
- **Who uses it.** The specific user type from Question 2.
- **Core experience.** The workflow from Question 3, written in present tense as if the product already works.

No capability lists, no state machines, no design rationale, no MVP wedge, no out-of-scope list. Those come later from `/product-author <feature>`.

### `docs/technical/stack.md`

YAML front matter: `title` and `summary` only. Under 200 lines.

Sections:
- **Architecture at a glance.** One paragraph: what runs where, how pieces communicate.
- **Stack.** The confirmed choices from Part 2, each with its one-sentence reasoning.
- **External services.** Anything the project depends on. If none, say so.
- **Scale expectations.** One line from Question 2.

### `docs/technical/patterns/core-rules.md`

This file must conform to fat-docs's pattern doc conventions. Read `docs/technical/patterns/overview.md` first to confirm the conventions haven't changed; the structure below reflects what they were at time of writing.

YAML front matter: `title: "Core coding rules"`, `summary`, `groups: ['global']`, `tags: ['code-style', 'rules']`. Under 100 lines.

Structure:

- **Scope.** One sentence: "Foundational language, type, and structural rules every file in this project follows. Override only with documented justification."
- **TL;DR (hard rules).** The numbered list from Part 3, reflecting the user's edits. Each rule starts with `MUST`, `NEVER`, `SHOULD`, or `PREFER` to match the priority-level convention in `patterns/overview.md`.
- **Status note** at the end: "Initial set from `/fresh-start`. Per-rule examples, anti-patterns, and verification queries will be added as patterns are exercised in real code."

### Update the patterns and technical overview indexes

Both `docs/technical/overview.md` and `docs/technical/patterns/overview.md` ship from fat-docs with a placeholder comment in their `## Documents` section (`<!-- ... appear here. -->`). After writing the new docs above, replace those placeholders with one-line entries:

- In `docs/technical/overview.md` — add: `- [Stack](stack.md) — language, runtime, libraries, services, and architecture-at-a-glance for this project.`
- In `docs/technical/patterns/overview.md` — add: `- [Core rules](core-rules.md) — foundational language, type, and structural rules every file follows.`

Leave both files' bodies otherwise untouched. Fat-docs owns the rest of those overview files.

## Part 5 — Run `index_docs` and hand off

Rebuild the docs search index via the `index_docs` MCP tool.

End with a short message to the user:

> "Fresh-start is done. Here's what you have:
>
> - `docs/product/overview.md` — what you're building, for whom, how they use it.
> - `docs/technical/stack.md` — your stack and architecture.
> - `docs/technical/patterns/core-rules.md` — the rules `/dev` follows when writing code.
>
> Read them and edit anything that doesn't match your intent. When you're ready, run `/dev` to scaffold the initial project files from your stack, or type `/fat-help` if you're not sure what to do next."

Do not auto-run `/dev`. Fresh-start is the boundary between *deciding* and *building*. Let the user cross it.

## Rules

- **No code.** Fresh-start produces docs only. Not one line of code, not even a `package.json`.
- **No feature-level questions.** If the user starts describing features, redirect: "We'll capture features one at a time with `/product-author` after fresh-start. For now, just the foundation."
- **Don't replicate `/product-author` or `/technical-author` in full.** Fresh-start is lighter on purpose. The other skills do the deep rigor later.
- **Don't negotiate patterns back and forth.** Show the list, accept edits, move on.
- **One stack proposal, not a menu.** If the user wants something different, they'll say so.
- **No time estimates, no phase announcements.** Just move through the questions.
- **No flattery.** "Great idea!" adds nothing. If something the user said has a real problem, name it once and move on.
